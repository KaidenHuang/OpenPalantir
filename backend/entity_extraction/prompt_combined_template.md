# 实体和关系提取任务

## 实体类型
- person: 人名
- organization: 组织（公司/机构/学校等）
- location: 地点（国家/城市/地理特征）
- event: 事件（会议/战争/活动等）
- concept: 抽象概念（政策/理论/思想等）
- other: 其他实体类型 （如 动物、植物、产品等）

## 输出JSON格式
最外层用entities和relationships作为key:
```json
{"entities": [{"n":"名称","t":"类型","c":次数,"bn":"别名","d":"描述"}], "relationships": [{"s":"主体","p":"关系","o":"客体","ot":"时间","d":"描述"}]}
```

## 规则
1. 实体名用标准全名，类型限上方6种值
2. `c`为int（出现次数），`bn`可选（别名）
3. 实体`d`为字符串，简要描述该实体在文本中的角色、身份或关键特征（25字以内），无明确信息时可省略
4. 关系的s和o必须在实体列表中
5. 关系用简洁中文动词或名词
6. 只返回纯JSON，无额外文本
7. 语言与输入文本一致
8. 相同名称+类型的实体去重
9. `ot`和`d`无值时为null
10. **方向一致性**：关系是有方向的，`s → p → o` 表示"主体s 对 客体o 有 p 关系"。对于同一对实体间的同一事实，只创建一条关系，选择最准确的 predicate 方向。例如"A是B的母亲"应写为 `{"s":"B","p":"母亲","o":"A"}`（B的母亲是A），不应同时写出 `{"s":"A","p":"儿子","o":"B"}` 这样方向矛盾的重复条目。
11. **避免冲突**：同一对实体间不能同时存在两条含义相反的关系（如"儿子"和"母亲"是同一关系的正反表达），只保留方向正确的一条。
